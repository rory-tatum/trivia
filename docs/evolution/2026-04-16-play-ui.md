# Evolution: play-ui

**Date**: 2026-04-16
**Feature ID**: play-ui
**Delivery status**: CLOSED — all 34 steps complete, all DES traces verified

---

## Feature Summary

The play-ui feature delivers the player-facing WebSocket interface for the trivia game at `/play`. Players can join a game room, register a team name, receive questions as the quizmaster reveals them, save draft answers, submit at round end, follow the ceremony with verdicts, and view round scores and final leaderboard — all driven by real-time WebSocket events with no polling and no manual navigation.

The feature also resolves four backend schema dependencies (DEP-01 through DEP-04) that extend the existing server to support the player room:

- **DEP-01**: `QuestionPublic` extended with `choices []string`, `is_multi_part bool`, and `media *MediaRef` — structural metadata copied by `StripAnswers()` with no answer exposure
- **DEP-02**: `ceremony_answer_revealed` broadcast scope extended to include `RoomPlay` in addition to `RoomDisplay`; payload extended with `verdicts []TeamVerdict`
- **DEP-03**: `submission_received` broadcast extended to `RoomPlay` — players see all teams' submission status in real time
- **DEP-04**: `RoundScoresPayload.Scores` and `GameOverPayload.FinalScores` restructured from `map[string]int` to typed arrays `[]ScoreEntry{team_id, team_name, round_score, running_total}`

**Scope**: 44 acceptance scenarios across 13 user stories (US-01 through US-13), plus 4 adapter integration scenarios (ADP-01 through ADP-03), plus infrastructure-failure scenarios (INF-01 through INF-06). Walking skeleton covers 6 stories end-to-end. All scenarios enabled by delivery close.

---

## Business Context

Players previously had no functional interface — the scaffold at `frontend/src/routes/Play.tsx` displayed "Status: connected" and the last event name only. This feature delivers the complete player game loop, enabling real multi-team trivia sessions. The emotional design was anchored on two personas: Priya Nair (anxious team captain responsible for submitting) and Jordan Kim (casual member who rejoins mid-round on a different device). Draft persistence and rejoin were prioritised in Release 1 over richer question types (Release 2) because the anxiety force around losing answers was identified as the dominant friction point.

## Key Decisions

### DISCUSS Wave (Luna, nw-product-owner — 2026-04-15)

- **[D1] Feature type: player-facing UI at /play** — No backend logic initiated from DISCUSS; four backend schema extensions flagged as DESIGN wave dependencies.
- **[D2] Walking skeleton strategy: new WS required** — Existing scaffold provided only WebSocket lifecycle; all UI components absent.
- **[D3] UX research depth: comprehensive** — Full emotional arcs for both personas; error paths, reconnection, and all 7 JTBD jobs covered.
- **[D4] JTBD: full analysis completed** — 7 jobs discovered (J1–J7); all 13 user stories trace to at least one job.
- **[D5] SSOT bootstrapping** — `docs/product/` did not exist; bootstrapped with journey YAML and jobs YAML.
- **[D6] Draft answers: localStorage is canonical** — `draft_answer` WS message is a secondary signal only; localStorage is the single source of truth for draft restoration.
- **[D7] Submission locking: UI-level enforcement** — After `submission_ack` with `locked: true`, answer fields are read-only client-side; server is authority for idempotency.
- **[D8] Release sequencing** — Walking Skeleton → Reliability → Question Types → Social Polish; rejoin prioritised above richer question types.

### DESIGN Wave (Titan, nw-system-designer — 2026-04-16)

- **[D1] DEP-01 resolved — QuestionPublic additive nullable fields** — Three optional fields with `omitempty`; `StripAnswers()` is the only sanctioned conversion point (DEC-010); answer fields never appear in transport.
- **[D2] DEP-02 resolved — ceremony broadcast to play + display with verdicts** — `handleCeremonyRevealAnswer` broadcasts to both `RoomDisplay` and `RoomPlay`; state machine enforces CEREMONY gate server-side.
- **[D3] DEP-03 resolved — submission_received broadcast to play room** — Single additional broadcast line; `SubmissionReceivedPayload` already carried required fields.
- **[D4] DEP-04 resolved — typed array with team names** — Breaking change to score payload shapes; host-ui and display consumers audited before shipping.
- **[D5] localStorage is canonical for draft answers** — Reaffirmed from DISCUSS; conflict resolution rule: snapshot fills gaps but does not overwrite non-empty localStorage entries.
- **[D6] Rejoin state routing via state_snapshot.state field** — Seven game states map to six distinct screens; no additional routing messages needed.
- **[D7] submission_received to play room is idempotent** — Submitter receives its own event; client identifies self by `team_id` and updates status list, not screen.
- **[D8] Multi-part answers encoded as JSON string in answer field** — Deferred cleaner `answers: []string` message field to Release 2.

### DISTILL Wave (Quinn, nw-acceptance-designer — 2026-04-16)

- **[DWD-01] Walking skeleton strategy C** — In-process `httptest.Server` with real production components; no mocks at the adapter boundary.
- **[DWD-02] Test scope: WebSocket protocol layer only** — React component behaviour (radio buttons, read-only fields, modal dialogs) is out of scope; belongs to React/Vitest unit tests.
- **[DWD-03] Host driver pattern reused for Given steps** — Host room WebSocket drives all game state preconditions.
- **[DWD-04] DEP-01 through DEP-04 all have explicit acceptance coverage** — Each backend change has at least one acceptance scenario as its acceptance gate.
- **[DWD-05] Error ratio gate met at 41.9%** — 18 of 43 focused scenarios are error/edge scenarios, exceeding the 40% mandate.
- **[DWD-06] Story traceability complete for all 13 user stories**.
- **[DWD-07] Upstream issues separated** — React-only UI concerns listed in `upstream-issues.md`; not represented in the feature file.
- **[DWD-08] No @kpi tag applied** — `docs/product/kpi-contracts.yaml` does not exist; KPI mapping provided informally in `test-scenarios.md`.

## Lessons Learned and Issues Encountered

1. **Walking skeleton step 01-01 is the highest-risk step** — Three DEPs wired simultaneously in a single step. This was correct architecturally (the DEPs are tightly coupled) but required the most implementation effort. Future high-risk walking skeletons should flag multi-DEP wiring explicitly in the roadmap to set time expectations.

2. **Breaking change to host-ui flagged in DISTILL and handled correctly** — DEP-02 required updating the host-ui acceptance test assertion from "play room does not receive ceremony_answer_revealed" to "play room receives ceremony_answer_revealed". This was flagged in `distill/wave-decisions.md` as a known breaking change, and was handled cleanly in step 01-01 without surprises. The upstream-issues mechanism worked as intended.

3. **Pre-existing infrastructure test failures are not a blocker but require tooling** — The WSL2 race detector timeout and missing Dockerfile caused failures in the mutation testing run. The `//go:build !mutation` approach is a viable workaround, but the underlying infra issues (Dockerfile, go-arch-lint config) should be addressed in a separate maintenance iteration rather than within feature delivery.

4. **Dead code cleanup caught in adversarial review** — An unused step implementation function (`thenCeremonyAnswerPayloadHasVerdicts`) and its orphaned import were identified during the adversarial review phase and cleaned up in the REFACTOR-L1-L4 step. The adversarial review phase caught this effectively.

5. **RED_UNIT skips are appropriate for adapter and step-impl-only steps** — 28 of 34 steps skipped RED_UNIT because existing helpers covered all domain behaviour and the step implementations were pure adapter/test-infrastructure code. This is correct per nWave methodology; the skip reasoning was documented in every case.

6. **DISTILL upstream-issues.md worked well for deferring React UI concerns** — Separating React component behaviour (radio buttons, read-only fields, modal dialogs) from the WebSocket protocol layer acceptance tests kept the feature file clean and focused. The upstream-issues mechanism is recommended for future UI features.
